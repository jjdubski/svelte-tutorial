🗺️ Execution Approaches

3 ways to organize all the work — new brainstormed features + existing open issues

1

Sequential by area

Pick an area, finish it, move on

Wave 1: Existing bug fixes (#15, #17, #19)
Wave 2: Template improvements (#25-#27)
Wave 3: Calendar redesign (#28-#29)
Wave 4: Query syntax
Wave 5: Keyboard shortcuts
Wave 6: PWA + icons
Wave 7: Settings route + themes
Wave 8: UX polish

Pro

  • Simple to track progress
  • Each wave independently testable

Con

  • Settings route (needed by themes) comes late
  • No momentum from quick wins
2

Foundation-first (recommended)

Build the settings hub early, then quick wins, then heavy lifts

Phase 1: Existing bug fixes + app icons + install prompt
Phase 2: Settings route + themes + notification prefs
Phase 3: Template improvements + calendar redesign
Phase 4: Query syntax + keyboard shortcuts
Phase 5: Transitions + drag polish + haptics + badge + background sync

Pro

  • Settings available early for theme work
  • Quick wins build momentum
  • Logical dependency ordering

Con

  • Templates/calendar delayed behind infrastructure
3

Existing issues first, then new features

Clear the backlog completely before starting new work

Batch 1: All 10 existing open issues (#15-#29)
└─ Bug fixes → Templates → Calendar → Svelte 5 refactor → Tests → Animations
Batch 2: All 6 brainstormed areas
└─ Any order within the batch

Pro

  • Clean break between old issues and new features
  • Reduces context switching

Con

  • Settings/themes delayed behind 10 issues
  • No quick wins for momentum

Which approach feels right? Or something in between?
